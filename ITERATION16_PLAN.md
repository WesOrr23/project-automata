# Iteration 16 — NFA → DFA Conversion (Plan)

## Context

After iter-15 lands save/load, the next biggest customer-impact item is the headline algorithm: subset construction. Every undergrad automata course assigns "convert this NFA to a DFA" exercises. A tool that doesn't do this is a toy.

Architecture review (Plan agent, this iteration) confirmed: eager subset construction over reachable subsets, with ε-closure handling, returning a side-table mapping new DFA states to the original NFA subsets. The engine's "engine owns IDs, UI owns labels" invariant stays intact.

Minimization, complement, equivalence — all natural follow-throughs — are deliberately punted to iter-17/18. Conversion alone is meaty enough.

## Goals

1. **`convertNfaToDfa(nfa) → Result<{ dfa, subsetMap }>`** in the engine. Pure function, fully tested.
2. **UI integration:** convert button in the tool menu, undoable (so the user can convert / revert / tweak / convert again as a side-by-side workflow at zero UI cost).
3. **Subset labels** in the converted DFA: `q2`, `{q0,q2,q5}`, `∅`. Generated by the UI from the subset map.
4. **Trap-state handling:** if `move(S, a) = ∅`, create (or reuse) a single trap state with self-loops on every symbol. Result is always a *complete* DFA.
5. **Sample fixtures** to demonstrate the feature (a textbook NFA in `src/data/`).
6. **Tests:** textbook fixtures, edge cases (pure-ε NFA, all-accept, single-state, no-accept), property test (random inputs accepted iff accepted by NFA).

## Non-goals

- Minimization (iter-17).
- Complement, equivalence, union, intersection (iter-17/18).
- Step-by-step animated construction (iter-20).
- True side-by-side two-canvas display (undo gives 95% of the value at 10% of the cost).
- Regex → NFA (iter-18).

## Algorithm — eager subset construction over reachable subsets

Standard textbook algorithm:

1. `S0 = ε-closure({nfa.startState})` — start subset.
2. Worklist seeded with `S0`. For each subset `S` taken from the worklist, for each non-ε symbol `a` in `nfa.alphabet`:
   - `T = ε-closure(union of move(s, a) for s in S)`
   - Add edge `S --a--> T`.
   - If `T` is unseen, mint a new DFA state for it and enqueue.
3. Stop when worklist is empty.
4. Accept-state set: every subset that contains at least one NFA accept state.

**Trade-off:** truly-lazy (compute on demand during simulation) saves memory but defeats the educational goal — users want to *see* the resulting DFA. Full powerset (eager all 2^n) wastes work on unreachable subsets and is exponential even when unnecessary. Reachable-only is what every textbook draws and what every visualizer uses.

`epsilonClosure` already exists in `src/engine/utils.ts` — reuse verbatim.

**Subset key for the seen-map:** sorted comma-joined string (e.g., `"0,2,5"`). Canonical and JS-Map-key safe.

## Engine API

```ts
// src/engine/converter.ts
export function convertNfaToDfa(nfa: Automaton): Result<ConversionOutcome>;

export type ConversionOutcome = {
  dfa: Automaton;
  subsetMap: Map<number, ReadonlySet<number>>; // newDfaStateId → original NFA state IDs
};
```

**Why a side table, not labels in the engine:** CLAUDE.md's "engine owns IDs, UI owns labels" invariant. Stuffing `"{q0,q2}"` into engine state breaks that. The subset map is structural metadata the UI consumes to build display labels via the existing `UIState.labels` map.

**New `EngineError` variants:**
- `'conversion-requires-nfa'` — passed a DFA. Refuse rather than no-op; user error worth surfacing.
- `'conversion-empty-alphabet'` — same precondition as simulation.
- `'conversion-no-start-state'` — defensive.

## State labeling (UI layer)

In `src/ui-state/labels.ts` (or wherever label generation lives):

- Single-element subset `{q2}` → `"q2"` (preserves familiarity)
- Multi-element `{q0,q2,q5}` → `"{q0,q2,q5}"` (sorted numerically)
- Empty (trap) → `"∅"`

Labels written into `UIState.labels: Map<number, string>` (already exists, no engine changes).

## Trap state — explicit

When `move(S, a) = ∅` for some `S, a`:
- If no trap state exists yet, mint one: empty subset, self-loops on every symbol, **non-accepting**.
- All subsequent "no transition" results route to the same trap state (single trap, not per-edge).
- Skip trap creation entirely if every reachable subset has a defined transition for every symbol (i.e., the conversion happens to be total).

**Why:** Result is a *complete* DFA. `isComplete()` validates. Converted DFA is immediately runnable with no surprise simulation behavior. "Implicit completeness" would make the result fail `isRunnable()` and confuse users who expect "convert" to produce a usable artifact.

## UX integration

- **Convert button** in the tool menu. Likely lives on a new "Operations" tab (alongside future "Minimize" in iter-17). For iter-16 alone, can also live on the existing Edit tab as a single new button.
- **Result replaces current automaton.** Conversion is undoable (existing iter-9 snapshot stack). User can convert / undo / tweak / convert again — that's the side-by-side workflow at zero new UI cost.
- **Auto-layout re-runs** on the new automaton (existing behavior).
- **Notification:** success ("Converted NFA to DFA — N states"); errors via `errorMessage(error)`.

**Side-by-side trade-off:** True side-by-side requires a second canvas, second simulator, second undo stack, layout coordination — a full iteration of its own. The undo-as-toggle pattern is dramatically cheaper. Real side-by-side becomes its own iteration if user demand emerges.

## Phase order

### Phase 1 — Engine
1. Add `EngineError` variants (`conversion-requires-nfa`, `conversion-empty-alphabet`, `conversion-no-start-state`) to `src/engine/result.ts` + `errorMessage` cases.
2. Create `src/engine/converter.ts`: `convertNfaToDfa` + `ConversionOutcome` type.
3. Reuse `epsilonClosure` from `utils.ts`.
4. Export from `src/engine/index.ts`.

### Phase 2 — Tests (TDD: write fixtures first)
5. Create `src/engine/__tests__/converter.test.ts`.
6. Fixtures:
   - **Textbook**: Sipser §1.2 example (NFA recognizing strings ending in 01) → known DFA. Assert exact subset map.
   - **Already-deterministic NFA**: NFA that's also a DFA. Conversion = structural identity (same language, isomorphic up to renumbering). Assert via accepts() equivalence over sampled inputs.
   - **Pure ε-NFA**: only ε-transitions. Verify start subset is full ε-closure; resulting DFA has only trap loops.
   - **All-accept NFA**: every reachable subset is accepting.
   - **Single-state NFA**: degenerate; output is single-state DFA (plus possibly trap).
   - **No-accept NFA**: zero accept states. Empty language.
   - **Trap creation**: NFA with missing transitions → DFA contains exactly one trap state with self-loops on all symbols. NFA with total transitions → no trap state.
7. **Property test**: for ~20 random short inputs, `accepts(nfa, w) === accepts(dfa, w)`.

### Phase 3 — UI integration
8. Add a `Convert to DFA` button in the tool menu. Disabled when `automaton.type === 'DFA'`.
9. Wire to a new `convertCurrentAutomaton` action in `useAutomaton.ts` (or App-level handler). Push onto the undo stack.
10. After successful conversion, populate `UIState.labels` from the subset map.
11. Show a notification on success/failure.

### Phase 4 — Sample fixture
12. Add `src/data/sample-nfa.json` (a textbook NFA). Wire it into the New menu (or wherever sample loading lives, post-iter-15).

### Phase 5 — End-to-end test
13. RTL test: load sample NFA, click Convert, assert canvas now shows DFA + accept states match expected, undo restores NFA.

## Success criteria

- `convertNfaToDfa(sample-nfa)` produces a DFA whose `accepts(w)` matches the NFA's `accepts(w)` for all tested strings.
- DFA states are labeled with their NFA subsets.
- Trap state is present iff the conversion required it.
- UI: convert button works, result is undoable, notifications surface success/failure.
- Test count: **~430 → ~470**.
- `tsc --noEmit` clean.

## Risks

- **Subset explosion for adversarial NFAs.** A pathological N-state NFA can produce 2^N reachable subsets. Mitigation: hard cap at 1000 generated subsets; return `Result.err('conversion-too-large')` if exceeded. (Add a fourth error variant.)
- **Auto-layout for large converted DFAs.** dagre handles ~50 nodes well; beyond that, layout slows. Mitigation: same hard cap at 1000 protects layout too.
- **Subset labels overflow node width.** A label like `{q0,q1,q2,q3,q4,q5}` is wide. Mitigation: truncate at length 30 with `…`; tooltip shows full label.
