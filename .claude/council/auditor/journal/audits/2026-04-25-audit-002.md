## New code-level observations the architect's review may have missed

**F8 — `exactOptionalPropertyTypes` flip widened component prop types codebase-wide, not just `notifications/types.ts`.**

The architect's iter-11 close-out flagged `notifications/types.ts` for "widening optional fields to `T | undefined` — re-admits the behavior the flag was meant to police." That callout produced commit `5cba947` which fixed *only* notifications/types.ts.

But `git diff 3607000~1 3607000 -- src/components/AutomatonCanvas.tsx` shows the same widening applied to ~12 props on `AutomatonCanvas` alone, plus comparable widening on `StateNode.tsx` (~7 props verified) and likely more components. Quick grep of `: .* | undefined` in non-test source returns 80+ matches concentrated in component prop types.

The `exactOptionalPropertyTypes` audit pass that the flag was meant to force happened in *one direction* (loosen everything) for components and was only tightened at the architect-flagged site. The TS-reviewer's `strict-flags.md` predicted the audit *should* "decide whether `T | undefined` ... or omit-only is intended" — that decision was effectively skipped by reflexively appending `| undefined` to widely-used component prop types.

**Severity**: not a regression of correctness — the flag is enabled and types compile. But the *intent* of the flag (force a decision) was satisfied at one site and bypassed at the rest. The TS-reviewer should be spawned on this.

**F9 — `useSimulation.ts` jump-to-step silently drops engine `Result` errors.**

In `jumpTo`, around the step loop: `if (!stepResult.ok) break;` — no notification, no log. The comment justifies it: *"if it errs (e.g. DFA dead-end on a stale automaton) we stop advancing and surface what we got so far rather than crashing the reducer."* This is deliberate degradation, not an oversight, but the architect's `result-type-error-model.md` "What to look for in diffs" calls out: *"Callsites that ignore the `Result` (no `if (!result.ok)` branch) — silent failures."* This callsite *has* the branch but the branch silently drops the error.

Worth a one-line clarification in `result-type-error-model.md` distinguishing "silently drop" from "ignore," or acknowledging the simulator's degradation pattern as an accepted exception.

**F10 — `useKeyboardScope` text-input filter has a subtle interaction with capturing scopes.**

If a modal opens while focus is in an `<input>` outside the modal, the modal's Escape handler will not fire while that input is focused (because `isTextInputFocused()` returns true and the listener exits early). Depends on whether modals take focus on open. Worth a test scenario in `useKeyboardScope.test.ts`.

## Patterns observed (meta)

**P5 — Single-agent close-outs leave the other agents' knowledge bases stale.**

Iter-11 was reviewed only by the architect. TS-reviewer and QA-reviewer were not invoked, so their knowledge bases are now stale on iter-11-relevant facts (F3, F4, F6). For iterations whose changes touch a domain (`exactOptionalPropertyTypes` is squarely TS territory; new test files are squarely QA territory), the orchestrator should spawn the affected reviewer for at least a "sweep my knowledge" pass.

This is a structural pattern, not a one-time miss. `_orchestrator.md` "Sequencing Rules of Thumb" mentions "spawn agents whose domains conflict" but doesn't say "spawn the agent whose knowledge is about to go stale."

**P6 — The journal-to-knowledge correction loop is now working; the close-out-to-knowledge loop is not.**

Audit-001's P1 (correction visible to auditor, invisible to agent on next read) is fixed structurally — `data-shape-facts.md` proves the loop closes. But the *iter-11 close-out* had two writer-level concerns. Both were fixed in code. Neither resulted in the architect's `keyboard-scope-stack.md` getting its "Known partial migration" section cleaned up after the migration *was* completed (F1). The "code fix → knowledge cleanup" path is the new soft spot.

**P7 — The architect is the heaviest writer in the council; this concentrates risk.**

Three of the architect's eight knowledge files were touched in iter-11 (two new, one updated). TS, QA, and Security all had zero touches. Partially structural (the architect is the right reviewer for refactor work like iter-11), but reinforces P5.

**P8 — `verified-as-of` is being used inconsistently.**

Some files use commit hashes (`52bdb8e`, `2cf5e42`), some use iteration markers (`iteration-11`). Mixed conventions reduce the auditor's ability to programmatically detect staleness. Recommend `_conventions.md` standardize: use commit hash whenever possible; use iteration markers only when work spans many commits and the hash would be misleading.
