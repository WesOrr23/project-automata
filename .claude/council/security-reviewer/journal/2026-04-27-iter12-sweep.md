## Findings

### F1 — Image export XSS via SVG state labels (severity: NONE / verified safe)

User-controlled state labels and transition symbols flow into SVG via React text children (`<text>{label}</text>` in `StateNode.tsx`, `<text>...</text>` in `TransitionEdge.tsx`). There are no places where labels/symbols/descriptions are interpolated into SVG attributes (`d=`, `style=`, `transform=`, `title=`).

Empirically verified with jsdom: a state label of `<script>alert(1)</script>` becomes a DOM text node with that literal string, `cloneNode(true)` preserves the text node, and `XMLSerializer().serializeToString()` re-encodes it as `&lt;script&gt;alert(1)&lt;/script&gt;` in the output. The exported `.svg` file opened in a browser shows the literal characters as visible text inside the FA — no script execution. Same for `<img src=x onerror=...>`. PNG path is even safer (canvas rasterization disposes of the DOM).

**No attacker action possible.** Calling out so the next reviewer doesn't re-investigate.

### F2 — `parseAutomataFile` prototype-pollution and DoS surface (severity: low)

The parser does direct property reads (`obj.kind`, `obj.formatVersion`, etc.), explicit object construction (`{ name: meta.name, ... }`), and `new Set(...)` reconstruction. No `Object.assign`, no spread of arbitrary keys, no recursive merge. A `__proto__` key in the input JSON would either be a no-op (modern JSON.parse semantics) or sit harmlessly as an own property that's never read. Strict shape-checking rejects anything not matching the `kind: "automata-file"` discriminator before any field extraction. This is correctly defensive.

**Minor gap**: there's no input size cap. A 100 MB JSON file would be passed to `JSON.parse`, then walked entirely. Realistic concern is low — browser memory will tap out and the user's tab will hang, not the worst-case outcome for a static educational app. If iter-19+ ever adds drag-drop or a network share path, a `MAX_FILE_BYTES` check at the parser entry becomes worth adding. Not a blocker for iter-12.

### F3 — `description` and notification `detail` rendering (severity: NONE / verified safe)

Both render via JSX text children / controlled-input `value` prop. React escapes both. No `dangerouslySetInnerHTML` anywhere in the codebase. No `innerHTML`/`outerHTML`/`document.write` in production code (only in test cleanup).

### F4 — localStorage keys and quotas (severity: low)

- All three keys carry the `automata-` prefix — collision risk on shared origins is realistic for "students opening multiple FA tools on a school server" but acceptably unlikely. The `-v1` suffix on the latter two is a genuine forward-compat plus.
- All three writers wrap `setItem` in try/catch — quota and private-browsing failures degrade gracefully (debug flag resets, tour reappears, recents drop oldest).
- `recentsStore.ts` has explicit caps: `MAX_ENTRIES=10`, `MAX_TOTAL_BYTES=1_000_000`, `MAX_PER_ENTRY_BYTES=250_000` with FIFO eviction. Good discipline.
- The recents `snapshot` field stores serialized `AutomataFile` content. **On reopen it's run through `parseAutomataFile` again** — defense-in-depth is correct, even hand-edited localStorage content can't bypass the validator.
- **Sensitive content**: the `description` free-text field is persisted into the recents snapshot. On a shared machine, a previous user's notes are visible to the next user via the recents list. This is a privacy concern (mild — the app is educational, not a notes app), not a security one. Worth a single sentence in the attack-surface knowledge file so it's documented.
- Keys themselves contain no sensitive data, just literal flag strings (`'on'/'off'`, `'seen'`).

### F5 — File adapter (severity: NONE / verified safe)

Save uses `Blob` + `URL.createObjectURL` + `<a download>` — same-origin blob URL, no exfiltration. Open uses synthesized `<input type="file" accept=".json,application/json">` + `FileReader.readAsText`. No drag-drop handler, no network call. Filename (`file.name`) is browser-attested, not user-typed; it flows into notification titles (escaped) and the recents `name` (also escaped on render).

The `accept=".json,application/json"` attribute is a hint, not enforcement — a user can still pick any file. But the parser rejects non-JSON via the `kind` discriminator, so any non-FA file is gracefully refused.

### F6 — Live SVG transform mutation race (out of scope — architect's domain)

`buildExportSVGString` calls `removeAttribute('transform')` on the live SVG, calls `getBBox()`, then restores. No try/finally — if `getBBox()` throws, the live canvas is left without the user's pan/zoom. This is a UX-correctness/architecture concern, not a security finding (no untrusted data involved). The architect already flagged it.
